# FHIR Packaging Guide

This guide walks you through assembling a complete, well-formed FHIR Package with `rh-packager`,
exercising every built-in processor in a realistic clinical example. You'll go from an empty
directory to a publishable `.tgz` FHIR Package in nine steps.

**Prerequisites:**
- `rh` installed (see [just install](../../justfile) or `cargo install --path apps/rh-cli`)
- FHIR R4 core package cached locally — run `rh download hl7.fhir.r4.core#4.0.1` once
- Python 3 available for the extended shell hook examples

> **Reference docs:** This guide focuses on the *how*. For authoritative field-by-field
> reference see [`README.md`](README.md) and [`PROCESSORS.md`](PROCESSORS.md).

---

## 1. Project Setup

### Directory layout

Create a directory for your package. By the end of this guide it will look like this:

```
bp-profiles/
  package.json                          # FHIR package manifest
  packager.toml                         # Processor pipeline configuration
  ImplementationGuide.json              # IG resource (must stay in sync with package.json)
  StructureDefinition-bp-observation.md # Resource narrative (embedded into matching resource)
  fsh/
    BpObservation.fsh                   # FSH source — compiled by the fsh processor
  cql/
    BpCheck.cql                         # CQL library — compiled by the cql processor
  docs/
    overview.md                         # Standalone narrative — copied to package/other/
  scripts/
    stamp_date.py                       # Shell hook — stamps build date onto resources
    check_date.py                       # Shell hook — validates build date is present
  fhir-lock.json                        # Generated by: rh publish lock
```

### `package.json`

Every FHIR Package starts with a `package.json`. Create `bp-profiles/package.json`:

```json
{
  "name": "example.bp-profiles",
  "version": "0.1.0",
  "description": "Blood pressure observation profiles",
  "fhirVersions": ["4.0.1"],
  "dependencies": {
    "hl7.fhir.r4.core": "4.0.1"
  },
  "author": "Example Organization",
  "license": "CC0-1.0"
}
```

### `ImplementationGuide.json`

`rh-packager` requires an `ImplementationGuide` resource whose `packageId` and `version` match
`package.json`. Create `bp-profiles/ImplementationGuide.json`:

```json
{
  "resourceType": "ImplementationGuide",
  "id": "example.bp-profiles",
  "url": "http://example.org/fhir/bp-profiles/ImplementationGuide/example.bp-profiles",
  "version": "0.1.0",
  "name": "BpProfiles",
  "title": "Blood Pressure Profiles",
  "status": "draft",
  "packageId": "example.bp-profiles",
  "fhirVersion": ["4.0.1"]
}
```

> **Auto-populated at build time.** You do not need to maintain `dependsOn`,
> `definition.resource`, or `definition.page` manually. `rh publish build` derives them
> automatically:
>
> - `dependsOn` — from `package.json` dependencies (excluding the core FHIR package)
> - `definition.resource` — one entry per resource/example in the package
> - `definition.page` — one child entry per `docs/*.md` standalone narrative file,
>   with `title` parsed from the first `# Heading` in each file
```

### `packager.toml`

The pipeline configuration wires all built-in processors together. Create
`bp-profiles/packager.toml`:

```toml
# Shared FHIR packages cache (default: ~/.fhir/packages)
# packages_dir = "~/.fhir/packages"

[hooks]
# Processors run in this order before the core build stage.
before_build = ["fsh", "snapshot", "validate", "cql", "build-info"]

[snapshot]
# Uses the shared packages_dir to resolve base profiles.

[validate]
# Validates every resource against its declared profile.

[cql]
# Compiles .cql files and upserts Library resources.

[fsh]
# Compiles *.fsh files recursively. All config inferred from package.json.

[processors.build-info]
command = "echo \"Built: $(date -u +%Y-%m-%dT%H:%M:%SZ)\" >> \"$PACKAGER_OUTPUT_DIR/other/build-info.txt\""
```

> **Processor ordering matters.** `fsh` compiles source to FHIR JSON, `snapshot` expands
> differentials so `validate` can check fully-resolved resources, and `cql` can reference the
> validated profiles. Shell hooks run last for any post-processing.

## 2. Step 1 — FSH: Author Profiles with FHIR Shorthand

[FHIR Shorthand (FSH)](https://fshschool.org) is a compact language for authoring FHIR profiles.
The `fsh` built-in processor scans your source directory recursively for `*.fsh` files and
compiles them into FHIR JSON resources, making them available to all subsequent pipeline stages.

Create `bp-profiles/fsh/BpObservation.fsh`:

```fsh
Alias: $LNC = http://loinc.org

Profile: BpObservation
Parent: Observation
Id: bp-observation
Title: "Blood Pressure Observation"
Description: "A blood pressure panel with systolic and diastolic components."

* status MS
* code = $LNC#55284-4 "Blood pressure systolic and diastolic"
* subject 1..1 MS
* subject only Reference(Patient)
* component ^slicing.discriminator.type = #pattern
* component ^slicing.discriminator.path = "code"
* component ^slicing.rules = #open

* component contains
    systolic 1..1 MS and
    diastolic 1..1 MS

* component[systolic].code = $LNC#8480-6 "Systolic blood pressure"
* component[systolic].value[x] only Quantity
* component[systolic].valueQuantity.unit = "mmHg"
* component[systolic].valueQuantity.system = "http://unitsofmeasure.org"
* component[systolic].valueQuantity.code = #mm[Hg]

* component[diastolic].code = $LNC#8462-4 "Diastolic blood pressure"
* component[diastolic].value[x] only Quantity
* component[diastolic].valueQuantity.unit = "mmHg"
* component[diastolic].valueQuantity.system = "http://unitsofmeasure.org"
* component[diastolic].valueQuantity.code = #mm[Hg]
```

When the pipeline runs, the `fsh` processor compiles this into
`StructureDefinition-bp-observation.json` and injects it into the build. No separate `rh` command
is needed — it runs automatically as part of `rh publish build`.

## 3. Step 2 — Snapshot: Expand Profile Differentials

FHIR profiles are authored as *differentials* — only the changes from the base resource. Many
tools (including validators and CDS engines) require the fully-expanded *snapshot* form. The
`snapshot` built-in processor expands every `StructureDefinition` in the pipeline that has a
differential but no snapshot.

The `[snapshot]` section in `packager.toml` has no required fields. It resolves base profiles
from the local FHIR packages cache (`~/.fhir/packages` by default, or the top-level
`packages_dir`).

After the snapshot processor runs, `StructureDefinition-bp-observation.json` will contain a
complete `snapshot.element` array alongside its `differential.element`. All downstream processors
(including `validate`) work with the fully-expanded form.

To override the packages cache for snapshot only:

```toml
[snapshot]
packages_dir = "/custom/path/.fhir/packages"
```

## 4. Step 3 — Validate: Enforce Conformance

The `validate` processor runs `rh-validator` over every FHIR resource in the package, checking
conformance against the base FHIR R4 specification and any declared profiles.

The `[validate]` section accepts an optional `packages_dir` override:

```toml
[validate]
packages_dir = "~/.fhir/packages"   # optional — inherits top-level default
```

**Passing build output:**

```
[validate] Validating 3 resource(s)...
  ✓ StructureDefinition/bp-observation
  ✓ ImplementationGuide/example.bp-profiles
  ✓ ValueSet/bp-codes
[validate] All resources valid.
```

**Validation failure — pipeline aborts:**

If a resource has a conformance error, the pipeline halts immediately and no output is written:

```
[validate] Validating 3 resource(s)...
  ✗ StructureDefinition/bp-observation
    ERROR: Constraint failed: sdf-1 (must have either a differential or snapshot)
[validate] Validation failed — aborting pipeline.
error: processor 'validate' exited with errors
```

Fix the error in your source, then re-run `rh publish build`.

## 5. Step 4 — CQL: Compile Clinical Logic

The `cql` built-in processor finds all `.cql` files in your source directory, compiles them to
ELM (Expression Logical Model), and upserts a FHIR `Library` resource for each library into the
package.

Create `bp-profiles/cql/BpCheck.cql`:

```cql
library BpCheck version '0.1.0'

using FHIR version '4.0.1'

include FHIRHelpers version '4.0.1' called FHIRHelpers

codesystem "LOINC": 'http://loinc.org'

code "BP Panel": '55284-4' from "LOINC" display 'Blood pressure systolic and diastolic'

context Patient

define "Blood Pressure Observations":
  [Observation: "BP Panel"] O
    where O.status in { 'final', 'amended' }

define "High Systolic Readings":
  "Blood Pressure Observations" O
    where exists (
      O.component C
        where C.code ~ "BP Panel"
          and (C.value as Quantity).value > 140
    )

define "Has Hypertensive Reading":
  exists "High Systolic Readings"
```

The `[cql]` section in `packager.toml` accepts an optional `packages_dir` override:

```toml
[cql]
packages_dir = "~/.fhir/packages"   # optional
```

After the CQL processor runs, a `Library-BpCheck.json` resource is upserted into the package.
It contains the compiled ELM as a base64-encoded `content` attachment alongside the original CQL
source, ready for use by CDS Hooks engines or FHIR Measure evaluators.

## 6. Step 5 — Shell: Custom Build Steps

The `[processors.<name>]` configuration lets you run any command as a pipeline hook — a bash
one-liner, a Python script, a Node.js tool, or anything else available on `PATH`.

### Environment variables

Every shell processor receives these environment variables:

| Variable | Description |
|---|---|
| `PACKAGER_SOURCE_DIR` | Absolute path to your source directory (`bp-profiles/`) |
| `PACKAGER_OUTPUT_DIR` | Absolute path to the output directory being assembled |
| `PACKAGER_WORKDIR` | Temporary working directory private to this processor invocation |
| `PACKAGER_PACKAGE_NAME` | Package name from `package.json` (e.g. `example.bp-profiles`) |
| `PACKAGER_PACKAGE_VERSION` | Package version from `package.json` (e.g. `0.1.0`) |
| `PACKAGER_FHIR_VERSIONS` | Comma-separated FHIR versions from `package.json` (e.g. `4.0.1`) |

### Resource exchange protocol

To read or modify FHIR resources from within a shell processor, use
`$PACKAGER_WORKDIR/resources/`. At the start of a `before_build` processor:

- Each resource in the current pipeline is written as `<ResourceType>-<id>.json`
- Your script reads, modifies, or creates files there
- After your script exits (with code 0), the packager reads back any changes

### Basic example — build info file

The `[processors.build-info]` entry in our `packager.toml` appends a timestamp to
`other/build-info.txt`:

```toml
[processors.build-info]
command = "echo \"Built: $(date -u +%Y-%m-%dT%H:%M:%SZ)\" >> \"$PACKAGER_OUTPUT_DIR/other/build-info.txt\""
```

A non-zero exit code from any shell processor aborts the pipeline immediately.

## 7. Extending the Pipeline — Python Hooks

For more complex post-processing, Python scripts are a natural fit. This section adds two Python
hooks that work together: `stamp_date.py` injects a build timestamp into every resource's
metadata, and `check_date.py` validates the stamp is present — acting as a quality gate.

### `scripts/stamp_date.py`

```python
#!/usr/bin/env python3
"""
Stamp a build-date tag onto every FHIR resource in the pipeline.
Reads from $PACKAGER_WORKDIR/resources/, writes back in place.
"""
import json
import os
import sys
from datetime import datetime, timezone
from pathlib import Path

resources_dir = Path(os.environ["PACKAGER_WORKDIR"]) / "resources"

if not resources_dir.exists():
    print("stamp_date: no resources directory found, nothing to stamp", file=sys.stderr)
    sys.exit(0)

build_date = datetime.now(timezone.utc).strftime("%Y-%m-%d")
tag = {
    "system": "http://example.org/tags",
    "code": "build-date",
    "display": build_date,
}

for path in resources_dir.glob("*.json"):
    resource = json.loads(path.read_text())
    meta = resource.setdefault("meta", {})
    tags = meta.setdefault("tag", [])
    # Remove any existing build-date tag before adding the fresh one.
    tags = [t for t in tags if not (t.get("system") == tag["system"] and t.get("code") == "build-date")]
    tags.append(tag)
    meta["tag"] = tags
    resource["meta"] = meta
    path.write_text(json.dumps(resource, indent=2))
    print(f"stamp_date: stamped {path.name} with build-date={build_date}")
```

### `scripts/check_date.py`

```python
#!/usr/bin/env python3
"""
Validate that every FHIR resource carries the build-date tag.
Exits non-zero (aborting the pipeline) if any resource is missing it.
"""
import json
import os
import sys
from pathlib import Path

resources_dir = Path(os.environ["PACKAGER_WORKDIR"]) / "resources"

if not resources_dir.exists():
    print("check_date: no resources directory found, skipping", file=sys.stderr)
    sys.exit(0)

errors = []
for path in resources_dir.glob("*.json"):
    resource = json.loads(path.read_text())
    tags = resource.get("meta", {}).get("tag", [])
    has_date = any(
        t.get("system") == "http://example.org/tags" and t.get("code") == "build-date"
        for t in tags
    )
    if not has_date:
        errors.append(path.name)

if errors:
    print("check_date: ERROR — missing build-date tag on:", file=sys.stderr)
    for name in errors:
        print(f"  - {name}", file=sys.stderr)
    sys.exit(1)

print(f"check_date: all {len(list(resources_dir.glob('*.json')))} resource(s) have build-date tag ✓")
```

### Wire the hooks in `packager.toml`

Replace the `[hooks]` section from Step 1 with the following to add both Python processors.
`stamp-date` runs in `before_build` so the tag is present before `validate` runs; `check-date`
runs in `after_build` as a final quality gate:

```toml
[hooks]
before_build = ["fsh", "snapshot", "stamp-date", "validate", "cql", "build-info"]
after_build  = ["check-date"]

[processors.stamp-date]
command = "python3 scripts/stamp_date.py"

[processors.check-date]
command = "python3 scripts/check_date.py"
```

### What failure looks like

If `check_date.py` finds a resource without the tag (e.g., because `stamp-date` was accidentally
removed from `before_build`), the pipeline aborts:

```
[processor:check-date] check_date: ERROR — missing build-date tag on:
[processor:check-date]   - StructureDefinition-bp-observation.json
[processor:check-date]   - Library-BpCheck.json
error: processor 'check-date' exited with code 1 — pipeline aborted
```

No output directory is written. Fix the hook configuration and re-run.

## 8. Building the Package

With all source files in place, build the package:

```sh
rh publish build bp-profiles/
```

Expected output:

```
[fsh]       Compiling 1 FSH file(s)...
[fsh]       ✓ StructureDefinition/bp-observation
[snapshot]  Expanding 1 StructureDefinition(s)...
[snapshot]  ✓ StructureDefinition/bp-observation
[stamp-date] stamp_date: stamped StructureDefinition-bp-observation.json with build-date=2024-11-01
[stamp-date] stamp_date: stamped Library-BpCheck.json with build-date=2024-11-01
[validate]  Validating 3 resource(s)...
[validate]  ✓ All resources valid.
[cql]       Compiling 1 CQL file(s)...
[cql]       ✓ Library/BpCheck
[build-info] Built: 2024-11-01T14:32:00Z
[check-date] check_date: all 3 resource(s) have build-date tag ✓
Building package...
✓ Wrote output/package/
✓ Packed example.bp-profiles-0.1.0.tgz
```

### Output layout

```
bp-profiles/
  output/
    package/
      package.json
      .index.json                              # Generated resource index
      ImplementationGuide-example.bp-profiles.json  # With dependsOn, definition.resource, definition.page auto-populated
      StructureDefinition-bp-observation.json  # Compiled from FSH, with snapshot and embedded narrative
      Library-BpCheck.json                     # Compiled from CQL
      examples/
        .index.json
        Observation-bp-example.json
      other/
        overview.md                            # Copied from docs/overview.md
        build-info.txt                         # Written by build-info shell hook
  example.bp-profiles-0.1.0.tgz               # The final FHIR Package tarball
```

The built `ImplementationGuide.json` will have `dependsOn`, `definition.resource`, and
`definition.page` fully populated — for example:

```json
{
  "resourceType": "ImplementationGuide",
  "id": "example.bp-profiles",
  "packageId": "example.bp-profiles",
  "version": "0.1.0",
  "fhirVersion": ["4.0.1"],
  "definition": {
    "resource": [
      {
        "reference": { "reference": "Library/BpCheck" },
        "name": "BpCheck",
        "exampleBoolean": false
      },
      {
        "reference": { "reference": "StructureDefinition/bp-observation" },
        "name": "Blood Pressure Observation",
        "description": "A blood pressure panel with systolic and diastolic components.",
        "exampleBoolean": false
      },
      {
        "reference": { "reference": "Observation/bp-example" },
        "exampleBoolean": true
      }
    ],
    "page": {
      "nameUrl": "index.md",
      "title": "Blood Pressure Profiles",
      "generation": "markdown",
      "page": [
        {
          "nameUrl": "other/overview.md",
          "title": "Blood Pressure Profiles",
          "generation": "markdown"
        }
      ]
    }
  }
}
```

The `.tgz` filename follows the convention `<package-name>-<version>.tgz` — matching the
`name` and `version` fields in `package.json`. This file is ready to publish to a FHIR
package registry or distribute directly.

## Next Steps

- **Lock canonical references** — run `rh publish lock bp-profiles/` to pin external canonical
  URLs to specific package versions in `fhir-lock.json`
- **Validate without building** — use `rh publish check bp-profiles/` to run validation without
  writing any output
- **Add narrative** — place a `.md` file matching a resource's filename stem (e.g.,
  `StructureDefinition-bp-observation.md`) and it will be embedded as `resource.text` during
  the build; place standalone docs (overview pages, etc.) in `docs/` and they are copied to
  `package/other/`
- **Build custom processors** — see [`PROCESSORS.md`](PROCESSORS.md) for the native Rust
  processor API and the full shell processor protocol reference
