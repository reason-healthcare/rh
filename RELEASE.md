# Release Guide: Rust Health (rh)

This is the **single source of truth** for releasing Rust Health across all distribution channels:

- **Crates.io** (Rust libraries and CLI)
- **NPM** (WebAssembly packages)
- **GitHub Releases** (pre-built binaries and install script)
- **Homebrew** (macOS/Linux)
- **Chocolatey** (Windows)
- **Docker** (containers)
- **Nix** (NixOS/flake)

---

## Release Overview

```
1. Prepare and validate locally
2. Bump versions
3. Run quality gates
4. Dry-run publish tests
5. Commit, tag, push
   ↓  (GitHub Actions triggered)
   ├→ Build binaries for all platforms
   ├→ Create GitHub Release (draft)
   ├→ Publish Rust crates to crates.io (manual)
   ├→ Update Homebrew tap (automatic)
   ├→ Update Chocolatey package (automatic)
   ├→ Push Docker image to GHCR (automatic)
   └→ Publish NPM packages (manual)
6. Review and publish GitHub Release draft
7. Update flake.nix (manual — cargoHash)
8. Verify all channels
```

## Step-by-Step Release Process

### 1. Prepare Your Code

Ensure all changes are merged to `main` and CI is passing:

```bash
git fetch origin
git checkout main
git pull origin main

# Verify CI status
gh run list --limit 1 --status completed
```

### 2. Determine the New Version

Check current versions:

```bash
just show-versions
```

Version format:

- **Stable**: `0.2.1`, `1.0.0`, `2.3.4`
- **Pre-release**: `0.3.0-beta.1`, `0.3.0-rc.1`, `0.3.0-alpha.1`

### 3. Bump All Versions

Update workspace version and all dependencies in one command:

```bash
just bump-version 0.3.0
```

This updates:
- `Cargo.toml` workspace version (`[workspace.package]`)
- `rh-validator`'s standalone version
- All internal `path` dependency version specifiers across every crate
- `flake.nix` version

**What `just bump-version` does NOT update** (must be done manually):
- `flake.nix` `cargoHash` — see Step 12
- `packages/*/package.json` version fields — see Step 11
- `pnpm-lock.yaml` — regenerated by `pnpm install`

Verify the changes:

```bash
git diff Cargo.toml Cargo.lock crates/*/Cargo.toml apps/*/Cargo.toml flake.nix
just show-versions
```

### 4. Run Quality Gates

```bash
# Format check and fix
just fmt

# Run full gate (format check, clippy, tests, audit)
just check
```

All checks must pass. If `cargo audit` reports issues:

- Fix dependencies with `cargo update`
- Or document exceptions in `audit.toml` with rationale

### 5. Test Dry-Run Publishes

Test crates.io publishing without uploading:

```bash
cargo publish --dry-run -p rh-foundation
cargo publish --dry-run -p rh-hl7-fhir-r4-core
cargo publish --dry-run -p rh-hl7-fhir-r5-core
cargo publish --dry-run -p rh-codegen
cargo publish --dry-run -p rh-cql
cargo publish --dry-run -p rh-fsh
cargo publish --dry-run -p rh-fhirpath
cargo publish --dry-run -p rh-vcl
cargo publish --dry-run -p rh-validator
cargo publish --dry-run -p rh-packager
cargo publish --dry-run -p rh-cli
```

All must exit 0. If any fail:

- Fix the issue
- Re-run `just check`
- Re-run dry-runs

### 6. Commit and Tag

```bash
# Stage all version changes
git add Cargo.toml Cargo.lock crates/*/Cargo.toml apps/*/Cargo.toml flake.nix

# Commit
git commit -m "chore: bump versions to 0.3.0"

# Tag release
git tag v0.3.0

# Push both
git push origin main --tags
```

This triggers the `release.yml` GitHub Actions workflow, which:

1. Builds binaries for macOS (Apple Silicon + Intel), Linux (musl), and Windows
2. Creates a **draft** GitHub Release with all artifacts
3. Pushes a Docker image to `ghcr.io/reason-healthcare/rh`

### 7. Publish to Crates.io

Wait for the tag push and monitor the workflow. Once tag CI completes:

```bash
# Publish in dependency order, waiting between each layer
cargo publish -p rh-foundation
sleep 30

cargo publish -p rh-hl7-fhir-r4-core
cargo publish -p rh-hl7-fhir-r5-core
cargo publish -p rh-codegen
cargo publish -p rh-cql
cargo publish -p rh-fsh
sleep 30

cargo publish -p rh-fhirpath
cargo publish -p rh-vcl
sleep 30

cargo publish -p rh-validator
sleep 30

cargo publish -p rh-packager
sleep 30

cargo publish -p rh-cli
```

If a publish fails with "dependency not found" or "crate version not found", wait 30–60 seconds and retry — the registry index can take a minute to propagate.

### 8. Verify Crates.io

All crates should be visible within minutes:

```bash
# Quick check
curl -s https://crates.io/api/v1/crates/rh-foundation | jq '.crate.newest_version'
curl -s https://crates.io/api/v1/crates/rh-cli | jq '.crate.newest_version'
```

Or verify individually:

```
https://crates.io/crates/rh-foundation
https://crates.io/crates/rh-hl7-fhir-r4-core
https://crates.io/crates/rh-hl7-fhir-r5-core
https://crates.io/crates/rh-codegen
https://crates.io/crates/rh-cql
https://crates.io/crates/rh-fsh
https://crates.io/crates/rh-fhirpath
https://crates.io/crates/rh-vcl
https://crates.io/crates/rh-packager
https://crates.io/crates/rh-validator
https://crates.io/crates/rh-cli
```

### 9. Monitor GitHub Release Build

The `release.yml` workflow creates a **draft** GitHub Release:

1. **Wait for all jobs to complete** (15–30 minutes typically)
2. **Review the draft** at: https://github.com/reason-healthcare/rh/releases
3. **Publish the draft** when ready

The release draft contains:

- Pre-built binaries for all platforms
- Install script (`install-rh.sh`)
- Release notes

### 10. Publish NPM Packages (WASM)

The WebAssembly packages must be published manually. **Do this after crates.io publishes succeed.**

Published packages:

- `@reasonhealth/fhirpath`
- `@reasonhealth/vcl`
- `@reasonhealth/cql`

(The private playground package is not published.)

#### Prerequisites

- Node.js 24 or later
- pnpm 10.11.0
- Rust stable with `wasm32-unknown-unknown` target
- `wasm-pack` installed
- npm account with publish access to `@reasonhealth`

```bash
pnpm install
rustup target add wasm32-unknown-unknown
cargo install wasm-pack
npm login
npm whoami
```

#### Update NPM Package Versions

Keep npm package versions aligned with the Rust workspace version. Update version in:

- `packages/fhirpath/package.json`
- `packages/vcl/package.json`
- `packages/cql/package.json`

Then run:

```bash
pnpm install
```

Commit `package.json` and `pnpm-lock.yaml` changes with the release version bump.

#### Build WASM

```bash
# Build all WASM packages
just wasm

# Or individually
cd crates/rh-fhirpath && just wasm
cd crates/rh-vcl && just wasm
cd crates/rh-cql && just wasm
```

#### Test Packages

```bash
cd packages
pnpm -r build
pnpm -r test
pnpm -r pack:dry-run
```

`pnpm -r pack:dry-run` confirms each publishable package includes:

- `dist/`
- `wasm/`
- `wasm-node/`
- `wasm-bundler/`
- `README.md`

#### Publish to NPM

```bash
# Stable release
npm publish --workspace @reasonhealth/fhirpath --access public
npm publish --workspace @reasonhealth/vcl --access public
npm publish --workspace @reasonhealth/cql --access public
```

For pre-release versions, use the `beta` tag so it doesn't become `latest`:

```bash
npm publish --workspace @reasonhealth/fhirpath --access public --tag beta
npm publish --workspace @reasonhealth/vcl --access public --tag beta
npm publish --workspace @reasonhealth/cql --access public --tag beta
```

> Do not use `--provenance` for local publishes; it requires a supported CI environment with OIDC.

#### Verify NPM Packages

```bash
npm view @reasonhealth/fhirpath version dist-tags
npm view @reasonhealth/vcl version dist-tags
npm view @reasonhealth/cql version dist-tags
```

Optionally install into a temporary project:

```bash
tmpdir="$(mktemp -d)"
cd "$tmpdir"
pnpm init
pnpm add @reasonhealth/fhirpath @reasonhealth/vcl @reasonhealth/cql
```

#### NPM Recovery

If a publish fails because the version already exists, do **not** republish over it — npm versions are immutable. Bump to a new patch or prerelease version and repeat the validation steps.

If a package was published with the wrong dist-tag:

```bash
npm dist-tag add @reasonhealth/fhirpath@<version> beta
npm dist-tag rm @reasonhealth/fhirpath latest
```

### 11. Update flake.nix cargoHash (Manual)

The `just bump-version` script updates the version in `flake.nix`, but the `cargoHash` must be updated separately because it depends on the actual cargo vendor output for the new version.

1. Attempt a Nix build to get the expected hash:

```bash
nix build
```

2. The build will fail with a hash mismatch error like:

```
error: hash mismatch in fixed-output derivation:
  specified: sha256-iLaaxz76QGRZ3U1eFdjLn0w9T7F8CJ9OqS/yTUjBjWY=
  got:    sha256-NEW_HASH_HERE
```

3. Update `cargoHash` in `flake.nix` with the `got` hash:

```bash
# Edit flake.nix line 27 with the new hash
```

4. Retry the build:

```bash
nix build
```

5. Commit the hash update:

```bash
git add flake.nix
git commit -m "chore: update flake.nix cargoHash for v0.3.0"
git push origin main
```

### 12. Verify the Complete Release

Once everything is published, verify all channels:

```bash
# Crates.io
cargo search rh-cli | head -5

# NPM
npm view @reasonhealth/fhirpath versions | tail -5

# GitHub Release
gh release view v0.3.0 --repo reason-healthcare/rh

# Docker
docker pull ghcr.io/reason-healthcare/rh:v0.3.0
docker run ghcr.io/reason-healthcare/rh:v0.3.0 --version

# Binary download test
curl -L https://github.com/reason-healthcare/rh/releases/download/v0.3.0/rh-x86_64-unknown-linux-musl.tar.gz | tar xzf - rh
./rh --version

# Nix (local flake)
nix shell github:reason-healthcare/rh -- rh --version
```

## Automatic Distribution Updates

After you publish the GitHub Release draft, the following happen **automatically**:

### Homebrew (macOS/Linux)

- `publish-homebrew.yml` workflow triggers on release publish
- Downloads macOS and Linux release binaries
- Computes SHA256 checksums
- Updates `Formula/rh.rb` in `reason-healthcare/homebrew-rh`
- Only runs for **stable** releases (skips `-beta` and `-alpha` tags)

**Status**: Automatic

**Secret**: `HOMEBREW_TAP_TOKEN` (PAT with `repo` scope)

**Manual fallback** (if needed):

```bash
git clone https://github.com/reason-healthcare/homebrew-rh
cd homebrew-rh
# Edit Formula/rh.rb with version, URLs, and SHA256 checksums
git add Formula/rh.rb
git commit -m "rh X.Y.Z"
git push
```

**User install**:

```bash
brew tap reason-healthcare/rh
brew install rh
```

### Chocolatey (Windows)

- `publish-choco.yml` workflow triggers on release publish
- Downloads Windows binary
- Updates `choco/rh.nuspec` version and `choco/tools/chocolateyinstall.ps1` checksum
- Packs and pushes `.nupkg` to chocolatey.org
- Only runs for **stable** releases (skips `-beta` and `-alpha` tags)

**Status**: Automatic

**Secret**: `CHOCOLATEY_API_KEY`

**Manual fallback** (if needed):

```powershell
cd choco
# Update version in rh.nuspec and checksum in tools/chocolateyinstall.ps1
choco pack
choco push rh.X.Y.Z.nupkg --source https://push.chocolatey.org/ --api-key <API_KEY>
```

**User install**:

```powershell
choco install rh
```

### Docker

- GitHub Actions automatically builds and pushes to ghcr.io
- Tagged as `ghcr.io/reason-healthcare/rh:v0.3.0` and `ghcr.io/reason-healthcare/rh:latest`
- Uses Docker Buildx with caching

**Status**: Automatic

**User usage**:

```bash
docker pull ghcr.io/reason-healthcare/rh:latest
```

### Install Script

- `scripts/install-rh.sh` is attached to each GitHub Release as an asset
- Auto-detects OS and architecture, downloads the correct binary
- Installs to `/usr/local/bin` by default

**Status**: Automatic

**Usage**:

```bash
# Install latest version
curl -fsSL https://raw.githubusercontent.com/reason-healthcare/rh/main/scripts/install-rh.sh | sh

# Install specific version
curl -fsSL https://raw.githubusercontent.com/reason-healthcare/rh/main/scripts/install-rh.sh | sh -s -- -v 1.0.0

# Install to custom directory
curl -fsSL https://raw.githubusercontent.com/reason-healthcare/rh/main/scripts/install-rh.sh | sh -s -- -d ~/.local/bin
```

### Nix (NixOS / Flakes)

- `flake.nix` in the repo provides a Nix package definition
- `just bump-version` updates the version field automatically
- **`cargoHash` must be updated manually** after each release (see Step 11)
- The nixpkgs package at `pkgs/by-name/rh/rh/package.nix` must be updated manually for each release

**Updating the nixpkgs package:**

After each release, submit a PR to [NixOS/nixpkgs](https://github.com/NixOS/nixpkgs) to update `pkgs/by-name/rh/rh/package.nix`.

**For the initial PR only (already done):**

1. Add yourself as a maintainer in `maintainers/maintainer-list.nix` in a separate commit before the package commit. Entry format:

   ```nix
   bkaney = {
     github = "bkaney";
     githubId = 5303;
     name = "Brian Kaney";
   };
   ```

2. Use commit message: `rh: init at X.Y.Z`

**For subsequent version bumps:**

1. Update `version`, `hash`, and `cargoHash` in `pkgs/by-name/rh/rh/package.nix`
2. Use commit message: `rh: OLD -> NEW` (e.g., `rh: 0.2.1 -> 0.3.0`)
3. Include a link to the upstream changelog or release notes in the commit body

**Computing hashes:**

The `hash` field (source tarball hash) changes with every version. Compute it with:

```bash
nix-prefetch-url --unpack https://github.com/reason-healthcare/rh/archive/refs/tags/vX.Y.Z.tar.gz
```

Or for `cargoHash`, build once and copy the `got:` hash from the error message (see Step 11).

**License format:**

The project is `MIT OR Apache-2.0`. In nixpkgs, this must be expressed as a **multi-line list** (nixfmt will flag a single-line list as improperly formatted):

```nix
license = with licenses; [
  mit
  asl20
];
```

**Not** `licenses.mit` (which would only declare MIT), and **not** `with licenses; [ mit asl20 ];` on a single line (nixfmt requires multi-line formatting for license lists).

**Other nixpkgs requirements:**

- `__structuredAttrs = true;` must be present (required by nixpkgs-vet for `pkgs/by-name/` packages)
- `cargoBuildFlags` and `cargoTestFlags` must use multi-line array format (nixfmt style)
- The maintainer list must reference `with maintainers; [ bkaney ]`, not `with maintainers; [ ]`

**NixOS AI policy:**

All commits must include an `Assisted-by:` trailer if LLM-based AI tools were used. Example:

```
rh: 0.2.1 -> 0.3.0

Assisted-by: Claude (Anthropic)
```

A `Co-authored-by:` trailer does **not** satisfy this requirement. See the [NixOS AI policy](https://github.com/NixOS/nixpkgs/blob/master/CONTRIBUTING.md#automationai-policy) for full details.

**Verifying locally:**

```bash
nix-build -A rh
```

**Reference: current nixpkgs package.nix template:**

## Generated FHIR Crate Release Policy

The R4 and R5 crates are generated artifacts and share the workspace version. Any codegen change that alters committed generated output must be treated as a generated-crate release change:

1. Regenerate both committed crates:

```bash
just regen-r4
just regen-r5
```

2. Verify no drift:

```bash
just regen-check
```

3. Build and test generated crates:

```bash
cargo test -p rh-hl7-fhir-r4-core
cargo test -p rh-hl7-fhir-r5-core
```

4. Include the generated output diff in the release commit and bump the workspace version before publishing.

If a codegen change is internal-only and `just regen-check` reports no diff, the generated crates do not need special release treatment beyond the normal workspace publish order.

## Releasing a Single Crate

Occasionally you may need to release one crate at a different version — for example, a patch fix in `rh-validator` without touching the rest of the workspace. Use the targeted recipes:

```bash
# Bump only rh-validator (leaves all workspace crates unchanged)
just bump-validator-version 0.2.1

# Bump only workspace crates (leaves rh-validator unchanged)
just bump-workspace-version 0.1.1
```

Then publish only the affected crate(s) and their dependents in dependency order. Dependents that weren't re-versioned do **not** need to be re-published unless their `version` specifier for the changed crate no longer satisfies the new version.

## Troubleshooting

### `cargo publish` fails with "already uploaded"

The version was already published. Bump the version and start from Step 3.

### `cargo publish` fails with "dependency ... does not have a version"

A workspace-internal `path` dependency is missing its `version` field. Add `version = "x.y.z"` matching the referenced crate's version and retry.

### `cargo publish` fails with "crate version not found"

A dependency was just published and the registry index hasn't propagated yet. Wait 30–60 seconds and retry.

### Dry-run passes but live publish fails

Usually a timing issue with registry sync. Wait and retry. Contact crates.io support if it persists.

### `cargo audit` finds vulnerabilities

Address the advisory before releasing. Run `cargo update` to pull in patched versions, or add an exception to `audit.toml` with documented rationale.

### GitHub Release draft doesn't appear

- Check the `release.yml` workflow status
- Verify binaries built successfully in CI
- Common issue: binary not stripped (check build logs)

### NPM publish fails with version conflict

npm versions are immutable. Bump to a new patch or prerelease version and repeat validation.

### NPM wrong dist-tag

Move tags explicitly:

```bash
npm dist-tag add @reasonhealth/fhirpath@<version> beta
npm dist-tag rm @reasonhealth/fhirpath latest
```

### Nix build hash mismatch

The `cargoHash` in `flake.nix` needs updating. Run `nix build`, copy the `got` hash from the error, update `flake.nix`, and retry. See Step 11.

### nixpkgs PR requires AI disclosure

NixOS requires an `Assisted-by:` git trailer on any commit that used LLM-based AI tools. See the Nix section above for format and link to the full policy. A `Co-authored-by:` trailer does **not** satisfy this requirement.
