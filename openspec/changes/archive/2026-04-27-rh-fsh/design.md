## Context

`rh` is a Rust FHIR toolkit with existing crates for FHIRPath (`rh-fhirpath`) and CQL (`rh-cql`), both using hand-written nom parsers over a custom `Span<'a>` type. The `rh-hl7_fhir_r4_core` crate provides FHIR R4 type definitions.

FSH (FHIR Shorthand) is the standard IG authoring language. The reference implementation (sushi) uses ANTLR4 in Node.js and becomes the primary bottleneck in large IG workflows. The performance problems are well-documented: serial processing, repeated StructureDefinition unfolding per rule, and GC pressure.

`rh-fsh` is a new crate in the same workspace. It targets the same use case — FSH → FHIR JSON — but is designed from the start for throughput.

## Goals / Non-Goals

**Goals:**
- Parse all FSH entity types (Profile, Extension, Logical, Resource, Instance, ValueSet, CodeSystem, Invariant, RuleSet, ParamRuleSet, Mapping) and all rule types into a typed Rust AST
- Resolve aliases and expand RuleSets (with cycle detection)
- Export FHIR JSON for all entity types, producing differential StructureDefinitions, ValueSets, CodeSystems, resource instances, and ConceptMaps
- Support parallel file parsing and parallel entity export via rayon
- Integrate as `rh fsh` CLI subcommand

**Non-Goals:**
- FHIR element-level validation (delegated to `rh-validator`)
- Snapshot generation (differential only; snapshot is expensive and better generated by the FHIR validator/IG Publisher)
- FSH language server protocol support
- Watch mode / incremental compilation
- Sushi config file or IG publication pipeline

## Decisions

### D1: nom + `Span<'a>` parser (consistent with CQL/FHIRPath crates)

The CQL crate demonstrates that a hand-written nom parser with a custom `Span<'a>` type (tracking line/column/offset) is sufficient for a production compiler. FSH's grammar is simpler than CQL's — it is line-oriented, every entity opens with a keyword, every rule line starts with `*`. This makes nom combinators an excellent fit with no need for a parser generator.

The `Span<'a>` type and its nom trait implementations from `rh-cql` are directly reused. The `lexer.rs` module provides FSH-specific keyword constants and helper parsers (whitespace, comments, multiline strings, cardinality, coded values, paths).

**Alternative considered**: A separate logos-based token stream + nom over the token stream. This would be slightly faster but introduces a build dependency and deviates from the established rh crate pattern. The performance gain from the lexer is small compared to the gains from parallelism and caching.

### D2: Pre-indexed `FhirDefs` for O(1) element lookups

Sushi's #1 export bottleneck is `ElementDefinition.unfold()` — it re-walks the entire StructureDefinition tree for every rule application. `rh-fsh` pre-processes all base R4 StructureDefinitions (from `rh-hl7_fhir_r4_core`) into a flat `HashMap<path, ElementSummary>` at startup. All element lookups during export become `HashMap::get()` — O(1) with no repeated tree traversal.

**Alternative considered**: Lazy per-SD unfolding with memoization. Rejected: adds complexity without meaningful advantage once the full index is built at startup.

### D3: rayon for parallel export

Each FSH entity is independent after RuleSet resolution. The export phase uses `rayon::par_iter()` to process all entities of each type in parallel. Shared read-only state (`Arc<FhirDefs>`) is safe to share across threads. Derived SD results are cached in a `DashMap` (or pre-serialized before the parallel phase).

**Alternative considered**: async/tokio. Rejected: export is pure CPU work, not I/O-bound. rayon's work-stealing thread pool is the right tool.

### D4: Differential-only StructureDefinition output

Generating snapshots requires resolving all inherited elements from the parent chain — expensive and redundant for most consumers (the FHIR validator and IG Publisher generate their own snapshots). `rh-fsh` emits differential-only SDs by default. Snapshot generation is left as a future opt-in flag.

### D5: Non-fatal error collection

Export errors on individual entities (e.g., unresolved parent, invalid path) are collected into `FhirPackage::errors` rather than aborting the entire compilation. This matches sushi's behavior and allows maximum output even with partial failures.

## Risks / Trade-offs

- **RuleSet expansion complexity** → Mitigation: topological sort with explicit cycle detection; cycles produce a clear `FshError::RuleSetCycle`.
- **ParamRuleSet parameter substitution** → Mitigation: implement as string-substitution on rule paths before parsing the expanded content; well-defined in the FSH spec.
- **FhirDefs startup cost** → Mitigation: `FhirDefs::r4()` is called once and wrapped in `Arc`; startup cost is amortized across all entities in a compilation run.
- **Missing FSH features** → Mitigation: focus on the 80% of FSH that covers 99% of real-world IGs; log unrecognized constructs as warnings, not errors.
- **Divergence from sushi output** → Mitigation: establish a test suite using known-good sushi outputs as golden files; track divergences explicitly.

## Open Questions

- Should `FhirDefs` support loading FHIR packages (`.tgz`) from a local package cache (e.g., `~/.fhir/packages`)? This would enable profiles that extend packages other than core R4. (Defer to a follow-on change.)
- Should the `rh fsh compile` command write output files to a directory structure matching sushi's output layout for drop-in compatibility? (Likely yes — needs investigation of sushi output conventions.)
