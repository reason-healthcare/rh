## Context

The rh workspace has 7 library crates and 1 CLI binary. None can be published to crates.io today because:
- Workspace `[workspace.package]` has placeholder `authors` and `repository` values
- `rh-hl7_fhir_r4_core` is generated code â€” its crate name is derived automatically from the FHIR package name (`hl7.fhir.r4.core` â†’ `hl7_fhir_r4_core`); there is no `rh-` prefix
- Path-only workspace-internal dependencies lack `version` specifiers (required by crates.io)
- `rh-fhirpath` and `rh-vcl` include `cdylib` in their `crate-type`, which prevents crates.io publication
- No beta version strategy has been established

The generated FHIR crate (`rh-hl7_fhir_r4_core/`) is owned by `rh-codegen`. Its `Cargo.toml` is regenerated by `rh codegen hl7.fhir.r4.core 4.0.1 --output ...`. Manual edits to this crate would be overwritten on next regeneration and must not be made.

## Goals / Non-Goals

**Goals:**
- Make all 8 packages publishable to crates.io at `0.x.0-beta.1` versions
- Add `--crate-name` option to `rh codegen` so the generated crate name can be overridden (enabling the `rh-` prefix on regeneration)
- Rename the generated crate from `hl7_fhir_r4_core` â†’ `rh-hl7-fhir-r4-core` by regenerating (never by hand-editing generated files)
- Fix workspace metadata placeholders
- Add `version` to all path dependencies
- Gate `cdylib` behind a `wasm` feature flag
- Establish publish ordering script/docs aligned with the dependency graph

**Non-Goals:**
- Stable 1.0 release
- Renaming or restructuring any other generated source files beyond Cargo.toml/lib-name
- Adding new runtime features or API surface

## Decisions

### 1. Override crate name via `--crate-name` in `rh codegen`

**Decision:** Add an optional `--crate-name <NAME>` argument to `CodegenArgs` (CLI layer) and thread it through `CrateGenerationParams` into `generate_cargo_toml`. When provided, this value is used as the crate name instead of the auto-derived `package.replace(['.', '-'], "_")`.

**Rationale:** The only place that writes `name = "hl7_fhir_r4_core"` into `Cargo.toml` is `generate_cargo_toml()` in `crate_generator.rs`. Adding a single optional field is minimal, non-breaking, and keeps the regeneration path clean. All other source files (`.rs` modules under `src/`) do not contain the crate name and are unaffected.

**Alternative considered â€” post-process sed/patch:** Patch the Cargo.toml after generation. Rejected: fragile, not idiomatic, and does not survive future regenerations.

**Alternative considered â€” manual edit once:** Edit the generated file and never regenerate. Rejected: violates the invariant that generated files are never hand-edited; regeneration would undo it silently.

### 2. Regenerate `rh-hl7_fhir_r4_core` with the new crate name

**Decision:** After adding `--crate-name`, run:
```
rh codegen hl7.fhir.r4.core 4.0.1 \
  --output crates/rh-hl7_fhir_r4_core \
  --crate-name rh-hl7-fhir-r4-core \
  --force
```
This regenerates only the Cargo.toml `name` and `[lib] name` fields in the generated crate. The crate directory itself stays `rh-hl7_fhir_r4_core/` (filesystem layout); only the published package name changes.

**Note:** Rust normalises hyphens to underscores for `use` statements. Downstream crates that `use hl7_fhir_r4_core::...` must be updated to `use rh_hl7_fhir_r4_core::...` and their `Cargo.toml` dependency key updated to `rh-hl7-fhir-r4-core`.

### 3. Workspace metadata inheritance

**Decision:** Fix `[workspace.package]` with real `authors`, `repository`, and `homepage` values. All per-crate Cargo.toml files that already use `workspace = true` fields inherit the fix automatically; no per-crate changes are needed for those fields.

### 4. Version strategy â€” `0.x.0-beta.1`

**Decision:** Set all crates to `0.1.0-beta.1` except `rh-validator` which will be `0.2.0-beta.1` (it has had more iteration). All workspace-internal `path` dependencies gain a `version = "0.1.0-beta.1"` (or `0.2.0-beta.1` for validator) alongside the `path` key.

**Alternative considered â€” `0.1.0` stable:** Rejected; the API is not sufficiently stabilised and beta signals early-adopter expectations correctly.

### 5. Gate `cdylib` behind a `wasm` feature and standardize WASM build tooling

**Decision:** Add a `wasm` feature to both `rh-fhirpath` and `rh-vcl`. The `[lib]` section keeps only `rlib` by default. A build script (`build.rs`) detects the `wasm` feature and emits the `cdylib` crate type at compile time. `wasm-pack` invocations always pass `--features wasm`. This is the same pattern used by the `wasm-bindgen` project itself.

**Concrete Cargo.toml shape (identical for both crates):**
```toml
[lib]
crate-type = ["rlib"]  # cdylib added at build time when wasm feature is active

[features]
wasm = ["dep:wasm-bindgen", "dep:js-sys", "dep:console_error_panic_hook"]

[package.metadata.wasm-pack.profile.release]
wasm-opt = ["-O4"]

[dependencies]
wasm-bindgen = { version = "0.2", optional = true }
js-sys = { version = "0.3", optional = true }
console_error_panic_hook = { version = "0.1", optional = true }
# Non-wasm deps remain unconditional

[target.'cfg(target_arch = "wasm32")'.dependencies]
web-sys = { version = "0.3", features = ["console"] }
```

**`build.rs` (identical for both crates):**
```rust
fn main() {
    if std::env::var("CARGO_FEATURE_WASM").is_ok() {
        println!("cargo:rustc-cfg=wasm_feature");
        // Emit cdylib crate type when wasm feature is enabled
        println!("cargo:rustc-link-arg=--crate-type=cdylib");
    }
}
```

> **Note:** An alternative that avoids `build.rs` is to keep `crate-type = ["cdylib", "rlib"]` as-is and accept it at crates.io â€” crates.io does not technically reject `cdylib` crates, but docs.rs cannot build documentation for them correctly, and the `cdylib` output is meaningless for downstream Rust users. The feature-gate approach is cleaner for the ecosystem.

**WASM build tooling â€” standardize on `just`:**
- Both crates use `justfile` (already the case). `rh-vcl/WASM_BUILD.md` incorrectly references "Makefile" and must be updated to match the justfile.
- Both justfiles are brought to an identical recipe set: `wasm`, `wasm-web`, `wasm-node`, `wasm-bundler`, `test-wasm`, `clean-wasm`, `dev-web`, `demo` (where applicable).
- All `wasm-pack build` invocations in both justfiles pass `--features wasm` to activate the `cdylib` crate type.
- `rh-fhirpath` already has `[package.metadata.wasm-pack.profile.release] wasm-opt = ["-O4"]`; add the same block to `rh-vcl/Cargo.toml`.

**Workspace-level coordination:**
Add a `wasm` recipe to the root `justfile` that delegates to both crates:
```just
# Build WASM packages for all wasm-capable crates
wasm:
    cd crates/rh-fhirpath && just wasm
    cd crates/rh-vcl && just wasm
```

**Rationale for consistency:** The two crates serve as the public WASM surface of the rh toolkit. Divergent tooling (one had a stale Makefile reference, different wasm-pack profiles, feature gates in different states) creates confusion for contributors and external users who want to build both. Uniform structure lets a single doc block cover both.

## Risks / Trade-offs

- **Downstream breakage from crate rename** â†’ Any code outside this repo that referenced `hl7_fhir_r4_core` by name will break. Mitigation: this is a pre-1.0 beta release; the breakage is expected and documented in the changelog.
- **Regeneration side-effects** â†’ Running `rh codegen --force` replaces the entire crate directory. Review the diff carefully before committing to confirm only Cargo.toml naming fields changed. Mitigation: run with `--force` only when intending a full regeneration; validate with `git diff crates/rh-hl7_fhir_r4_core/`.
- **`wasm` feature opt-in breaks existing WASM build commands** â†’ Anyone currently running `wasm-pack build` without `--features wasm` will not get a `cdylib` and the build will fail. Mitigation: update all justfile `wasm-pack build` invocations to include `--features wasm`; document in READMEs and CHANGELOG.
- **`build.rs` approach for `cdylib`** â†’ Using `build.rs` to influence crate type is a non-standard pattern. Mitigation: if it proves unreliable across Cargo versions, fall back to keeping `crate-type = ["cdylib", "rlib"]` and document the docs.rs limitation.

## Migration Plan

1. Add `--crate-name` to `rh codegen` (library + CLI, no breaking API change)
2. Fix `[workspace.package]` metadata
3. Update all crate versions to beta
4. Regenerate `rh-hl7_fhir_r4_core` using `--crate-name rh-hl7-fhir-r4-core --force`
5. Update downstream `Cargo.toml` dependency keys and `use` paths for the renamed crate
6. Add `wasm` feature to `rh-fhirpath` and `rh-vcl`; add `build.rs`; move wasm-bindgen/js-sys/console_error_panic_hook to optional deps under `wasm` feature
6a. Update both justfiles to pass `--features wasm` to all `wasm-pack build` invocations
6b. Add `[package.metadata.wasm-pack.profile.release]` to `rh-vcl/Cargo.toml`
6c. Update `rh-vcl/WASM_BUILD.md` to reference `just` (not `make`)
6d. Add `wasm` recipe to root `justfile`
7. Run `cargo publish` in dependency order: rh-foundation â†’ rh-hl7-fhir-r4-core â†’ rh-codegen â†’ rh-cql â†’ rh-fhirpath â†’ rh-vcl â†’ rh-validator â†’ rh-cli
8. Validate with `cargo publish --dry-run` before each real publish

**Rollback:** All changes are in source; reverting the branch restores the prior state. crates.io versions, once published, cannot be deleted but can be yanked.
